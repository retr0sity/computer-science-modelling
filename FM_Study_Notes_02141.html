FM1 — Program Graphs

Program Graphs & What They Model

A formal method is a mathematically precise technique for reasoning about programs. Testing can only show bugs exist; formal methods let you prove correctness for all inputs. Real examples: seL4 (verified bug-free OS kernel) and WebAssembly (designed with a formal semantics from day one).

1.1 Program Graphs ⭐ EVERY EXAM

Plain English

A program graph is a precise flowchart. Nodes = places a program can "be" (before loop, inside loop, done). Edges = moves the program makes, labelled with what they do (assignments or boolean tests).

Formal Definition

PG = (Q, q▷, q◀, Act, E)

• Q — finite set of nodes (control locations)
• q▷ ∈ Q — unique initial node
• q◀ ∈ Q — unique final node
• Act — set of actions (assignments and boolean tests)
• E ⊆ Q × Act × Q — finite set of directed edges (from-node, action, to-node)

Worked Example — Factorial Program ⭐

y := 1;
while (x > 0) {
    y := x * y;
    x := x - 1
}

Step 1 — Nodes: q▷ = start  |  q₁ = at while-test  |  q₂ = doing y:=x*y  |  q₃ = doing x:=x-1  |  q◀ = done

Step 2 — Edges:

(q▷) ──[y:=1]──▶ (q₁) ──[x≤0]──▶ (q◀) │ [x>0] ▼ (q₂) ──[y:=x*y]──▶ (q₃) ▲ │ └────[x:=x-1]────────┘

Formally: E = { (q▷,y:=1,q₁), (q₁,x>0,q₂), (q₂,y:=x*y,q₃), (q₃,x:=x-1,q₁), (q₁,x≤0,q◀) }

Key point: Boolean tests like x>0 and x≤0 are also "actions" on edges. They don't change variable values — they gate which edges can be taken. Control-flow keywords (while, if) disappear into the graph structure.
FM1 — Memory & Semantics

Memory, Semantic Functions & Determinism

1.2 Memory (Program State) ⭐

Plain English

A memory is a snapshot of all variable values — a table mapping each variable name to its current integer value.

Memory: σ : Var → Int  (assigns an integer to every variable)

Memory update: σ[x ↦ i] = σ except x now maps to i
σ[x ↦ i](y) = i    if y = x
σ[x ↦ i](y) = σ(y) if y ≠ x
Example: If σ = {x:12, y:5} then σ[y ↦ 7] = {x:12, y:7}

1.3 Semantics of Actions ⭐

Plain English

Each action says how it transforms memory. An assignment updates a variable. A boolean test either proceeds (memory unchanged) or blocks (undefined — edge cannot fire).

Semantic function: S[[·]] : Act → (Mem ↪ Mem)
Maps each action to a partial function on memories (partial = can be undefined).
Action αS[[α]](σ)Notes
x := aσ[x ↦ A[[a]](σ)]Always defined (no div-by-zero)
b when B[[b]](σ) = trueσ (unchanged)Guard passes; memory unmodified
b when B[[b]](σ) = falseundefinedEdge cannot be taken
Examples:
S[[y:=1]](σ) = σ[y↦1]
S[[x>0]](σ) = σ if σ(x)>0; undefined otherwise
S[[y:=x*y]](σ) = σ[y ↦ σ(x)×σ(y)]

1.5 Deterministic Systems ⭐

Plain English

A system is deterministic if at most one execution is possible for each starting memory. No memory should enable two different outgoing edges from the same node simultaneously.

Determinism condition: For all edges (q,α₁,q₁),(q,α₂,q₂) ∈ E with (α₁,q₁)≠(α₂,q₂):
dom(S[[α₁]]) ∩ dom(S[[α₂]]) = ∅
Guards like x>0 and x≤0 always have disjoint domains — exactly one fires for any x. This is why GCL guards ensure determinism.
FM1 — Operational Semantics

Configurations & Execution Steps

1.4 Configurations & Execution ⭐ EVERY EXAM

Plain English

A configuration is a snapshot of the running program: which node you are at (the program counter) and what all variables hold (the memory). An execution step fires an edge, moving to a new configuration.

Configuration: ⟨q; σ⟩ where q ∈ Q and σ ∈ Mem
Initial: q = q▷  |  Final: q = q◀

Execution step: ⟨q₀; σ⟩ ⟹α ⟨q•; σ'⟩ whenever
  1. (q₀, α, q•) ∈ E — edge exists
  2. S[[α]](σ) = σ' — action is defined and produces σ'

Reachability: ⟨q₀;σ⟩ ⟹* ⟨q•;σ'⟩ means reachable via 0 or more steps

Full Execution Trace — Factorial with x=2 ⭐

#ConfigurationAction firedResult
0⟨q▷; {x:2, y:0}⟩—start
1⟨q▷; {x:2,y:0}⟩ ⟹y:=1⟨q₁; {x:2,y:1}⟩
2⟨q₁; {x:2,y:1}⟩ ⟹x>0 (2>0 ✓)⟨q₂; {x:2,y:1}⟩
3⟨q₂; {x:2,y:1}⟩ ⟹y:=x*y → 2×1=2⟨q₃; {x:2,y:2}⟩
4⟨q₃; {x:2,y:2}⟩ ⟹x:=x-1 → 2-1=1⟨q₁; {x:1,y:2}⟩
5⟨q₁; {x:1,y:2}⟩ ⟹x>0 (1>0 ✓)⟨q₂; {x:1,y:2}⟩
6⟨q₂; {x:1,y:2}⟩ ⟹y:=x*y → 1×2=2⟨q₃; {x:1,y:2}⟩
7⟨q₃; {x:1,y:2}⟩ ⟹x:=x-1 → 1-1=0⟨q₁; {x:0,y:2}⟩
8⟨q₁; {x:0,y:2}⟩ ⟹x≤0 (0≤0 ✓)⟨q◀; {x:0,y:2}⟩ ✅
Result: Starting with x=2, program terminates with y=2 = 2! ✓
The operational semantics = the set of all complete execution sequences (all paths from q▷ to q◀).
FM2 — Guarded Command Language

GCL: Syntax & Translation to Program Graphs

GCL (Guarded Command Language) was designed by Dijkstra — a tiny language ideal for formal reasoning: small enough to define with complete mathematical precision, large enough to express all real programs.

2.1 GCL Syntax ⭐

Commands C:
  C ::= x := a        (assignment)
      | skip           (no-op)
      | C₁; C₂         (sequencing)
      | if GC fi       (conditional)
      | do GC od       (loop)

Guarded commands GC:
  GC ::= b -> C        (if b holds, execute C)
       | GC₁ [] GC₂    (choice — non-det. if both guards hold)
ConstructGCLMeaning
Assignmentx := aSet x to value of expression a
No-opskipDo nothing
SequenceC₁; C₂C₁ then C₂
Conditionalif b₁->C₁ [] b₂->C₂ fiExecute whichever branch's guard is true
Loopdo b->C odRepeat while b holds; exit when ¬b
Choice op[]Non-deterministic if both guards simultaneously true
// Factorial
y := 1;
do x > 0 -> y := x * y;
            x := x - 1
od

// Maximum of x, y stored in z
if x >= y -> z := x
[] y > x  -> z := y
fi

2.2 Translating GCL to Program Graphs ⭐ EVERY EXAM

Plain English

Each GCL construct has a systematic recipe. The translation is compositional: translate each subcommand, then glue the pieces together by sharing nodes.

Key function: edges(q₀ ↝ q•)[[C]] — the set of edges produced by translating command C between entry node q₀ and exit node q•.

GCL ConstructTranslation Rule
x := aAdd one edge: (q₀, x:=a, q•)
skipAdd one edge: (q₀, skip, q•)
C₁; C₂Fresh node q_mid; translate C₁: q₀↝q_mid, then C₂: q_mid↝q•
if b₁->C₁ [] b₂->C₂ fiTranslate each branch independently, both sharing entry q₀ and exit q•
do b->C odGuard-true branch loops back to q₀; guard-false edge (¬b) exits to q•

Worked Translation ⭐

Translate: y := 0; do x ≥ 0 -> y := y + x*x; x := x - 1 od

  • Outer structure = C₁; C₂. Introduce q₁ as midpoint between q▷ and the loop.
  • C₁ = y:=0: add edge (q▷, y:=0, q₁).
  • Loop body = y:=y+x*x; x:=x-1. Introduce q₂ (after 1st assignment) and q₃ (after 2nd).
  • Loop enter: (q₁, x≥0, q₂). Loop exit: (q₁, x<0, q◀).
  • Body: (q₂, y:=y+x*x, q₃) and (q₃, x:=x-1, q₁) — the loop-back edge.
(q▷) ──[y:=0]──▶ (q₁) ──[x<0]──▶ (q◀) │ [x≥0] ▼ (q₂) ──[y:=y+x*x]──▶ (q₃) ──[x:=x-1]──▶ ↑
FM3 — Deterministic Semantics

Deterministic GCL & Structural Induction

3.1 Making GCL Deterministic

The standard if ... fi is non-deterministic if two guards are simultaneously true. The deterministic variant adds "no previous guard held" to each branch. Branch bᵢ gets extra guard ¬(b₁ ∨ ... ∨ bᵢ₋₁) — it fires only when all earlier guards were false.

Example: if x≥y -> z:=x [] y>x -> z:=y fi
Guards x≥y and y>x are mutually exclusive for any distinct x,y — already deterministic.
But x≥y and y≥x overlap when x=y — that would be non-deterministic!

3.2 Structural Induction

To prove a property holds for all GCL programs, use structural induction — mirroring how programs are built inductively from atoms.

  • Base cases: prove for x:=a and skip.
  • Inductive step: assume for C₁, C₂; prove for C₁;C₂, if GC fi, do GC od.
FM4 — Program Verification

Hoare Logic: Specifying & Proving Program Correctness

Program verification = proving a program does what it should, for all inputs. Dijkstra: "Program testing can be very effective to show the presence of bugs, but it is hopelessly inadequate for showing their absence."

4.1 Predicates

Predicate: a set P ⊆ Mem of memories (a condition on variable values).

σ ⊨ P — σ satisfies P (σ ∈ P)
P ⊨ Q — P entails Q (P ⊆ Q — every memory in P is also in Q)
English conditionAs a predicate
x is positive{σ | σ(x) > 0}
x is even{σ | ∃y: σ(x) = 2y}
y = x factorial{σ | σ(y) = factorial(σ(x))}

4.2 Floyd-Hoare Triples ⭐ CORE TOPIC

Plain English

A Hoare triple is a mathematical contract for a program. It says: "If you start in a state where P holds, and the program terminates, then Q holds at the end." This is partial correctness — termination is separate.

⊨ {P} C {Q} is valid iff:
For every complete execution sequence ⟨q▷; σ⟩ ⟹* ⟨q◀; σ'⟩ of C:
   if σ ⊨ P   then   σ' ⊨ Q
TripleMeaning
{x=n ∧ n≥0} factorial {y=n!}Factorial program is correct
{true} x:=5 {x=5}After assigning 5, x equals 5
{x>5} x:=x+1 {x>6}Incrementing from >5 gives >6
⭐ Exam pattern: Given a program and specification, either prove the triple using proof rules (show each rule application), or find a counterexample execution to disprove it.
FM4 — Proof Rules

Hoare Logic Proof Rules ⭐ ALL MUST BE KNOWN

These rules let you prove triples structurally — one construct at a time. A valid proof is a tree; every leaf is an axiom.

skip — (no premises)
{Q} skip {Q}

skip does not change memory, so whatever held before still holds after.

assign — (no premises)
{Q[a/x]}   x := a   {Q}

Q[a/x] = "Q with every x replaced by a." This is backward reasoning: start from the postcondition and compute the required precondition.

seq {P} C₁ {R}      {R} C₂ {Q}
{P} C₁; C₂ {Q}

R is the "midpoint predicate." Choosing R well is the creative step.

cond {P ∧ b₁} C₁ {Q}   ...   {P ∧ bₙ} Cₙ {Q}
{P} if b₁->C₁ [] ... [] bₙ->Cₙ fi {Q}

Each branch verified independently, with its own guard added to the precondition.

cons (consequence) P ⊨ P'     {P'} C {Q'}     Q' ⊨ Q
{P} C {Q}

Allows strengthening the precondition or weakening the postcondition. Used when the other rules give a slightly different pre/post than needed.

loop (do-rule) P ⊨ I      {I ∧ b} C {I}      I ∧ ¬b ⊨ Q
{P} do b -> C od {Q}

I is the loop invariant. Three things to check: (1) I holds before the loop, (2) I is preserved by each iteration, (3) when loop exits (¬b), I implies Q.

Worked Example: Assignment Rule ⭐

Goal: Prove {x > 5} x := x+1 {x > 6}

  • Postcondition Q = "x > 6".
  • Apply [assign]: compute Q[x+1/x] = "x+1 > 6" = "x > 5".
  • [assign] gives us: {x>5} x:=x+1 {x>6} ✓

Goal: Prove {true} x:=5; y:=x+1 {y=6}

  • Apply [assign] to y:=x+1 with postcondition Q = "y=6". Substitute x+1 for x: Q[x+1/x] = "x+1=6... wait — here we assign to y, so substitute in the postcondition for y's new value. The postcondition "y=6" with y replaced by x+1 gives "x+1=6", i.e., "x=5". Result: {x=5} y:=x+1 {y=6}.
  • Apply [assign] to x:=5 with Q = "x=5". Q[5/x] = "5=5" = true. Result: {true} x:=5 {x=5}.
  • Apply [seq] with midpoint R = "x=5": conclude {true} x:=5; y:=x+1 {y=6} ✓
FM4 — Loop Invariants

Loop Invariants ⭐ HIGHEST EXAM FREQUENCY

Plain English

A loop invariant I is a condition that is true at the start of every single iteration. It "survives" each pass through the loop body. This lets us reason about a loop that runs a million times using just one logical statement.

The hard part: Finding a good invariant is the creative step — there is no algorithm for it. Checking the three conditions (once you have I) is mechanical. Practise on factorial, counting, and integer division.

Full Worked Example — Factorial ⭐

{ x = n ∧ n ≥ 0 }
y := 1;
do x > 0 ->
    y := x * y;
    x := x - 1
od
{ y = n! }

Invariant I: y · x! = n! ∧ x ≥ 0

  • P ⊨ I: Before loop (after y:=1): y=1, x=n. Does y·x! = n!? 1·n! = n! ✓   x≥0? n≥0 ✓
  • {I∧b} C {I}: Assume y·x!=n! and x>0. After y:=x*y: new_y=x·y. After x:=x-1: new_x=x-1. New_y·(new_x)! = x·y·(x-1)! = y·x! = n! ✓   new_x=x-1≥0 (since x≥1) ✓
  • I∧¬b ⊨ Q: Loop exits when x≤0. With I: x≥0. So x=0. Then y·0!=n! → y·1=n! → y=n! ✓
Proof outline: 
{ x = n ∧ n ≥ 0 }
y := 1;
{ y · x! = n! ∧ x ≥ 0 }    ← invariant established
do x > 0 ->
    { I ∧ x > 0 }
    y := x * y;
    x := x - 1
    { I }                    ← invariant preserved
od
{ y · x! = n! ∧ x = 0 }
{ y = n! }                   ← postcondition ✓

5.2 Non-Interference ⭐

Plain English

Non-interference: run the same program twice with different secret values but the same public inputs. If you always get the same public outputs, secrets cannot leak.

u interferes with v if there exist memories σ₁, σ₂ such that:
1. σ₁(v) = σ₂(v)   (v starts equal in both runs)
2. ⟨q▷,σ₁⟩ ⟹* ⟨q◀,σ'₁⟩   and   ⟨q▷,σ₂⟩ ⟹* ⟨q◀,σ'₂⟩
3. σ'₁(v) ≠ σ'₂(v)   (v ends differently)

If u does not interfere with v: final value of v is independent of initial value of u.
ProgramInterference?Secure?
b := aYes — b changes when a changes❌ Insecure (secret a leaks to public b)
a := bNo — b unchanged✓ OK
b := 0No — b always 0✓ Secure
if a=0->b:=1 [] true->b:=0 fiYes — b's value reveals whether a=0❌ Implicit flow!
Implicit flows: Even without directly assigning a secret to a public variable, a conditional whose guard depends on a secret leaks information via which branch was taken. This is subtle and the most common exam trick.

Two-Copy Verification Technique ⭐

To prove a does not interfere with b:

  • Create two copies of the program — rename every variable x to x₁ in copy 1, x₂ in copy 2.
  • Form the product program P₁×₂.
  • Prove the Hoare triple {b₁=b₂} P₁×₂ {b₁=b₂}: if both copies start with the same b value, they end with the same b value.
  • If this holds, b cannot reveal anything about a. ✓   Tool: Chip at chip-pv.netlify.app/chip

5.3 Security Type Systems ⭐

Plain English

A security type system enforces non-interference statically. Variables get a security level (H = secret, L = public). The rule: information can only flow upward in the lattice.

H (High / secret / private) ↑ ← information can flow upward: L→H is OK L (Low / public)

FM6 — Concurrency

Parallel Composition & Race Conditions

Parallel Composition

Two programs P₁ and P₂ running concurrently are modelled by the product of their program graphs. A combined configuration is ⟨q¹, q²; σ⟩ — one node per process, plus shared memory. Either process can take a step at any time: interleaved semantics.

Execution step (parallel): Process i takes its step (q^i_o, α, q^i_•) if the edge exists and S[[α]](σ) = σ'. All other processes stay at their current nodes.
Race condition example:
P₁: if balance≥pay_Bob -> balance:=balance-pay_Bob fi
P₂: if balance≥pay_Charlie -> balance:=balance-pay_Charlie fi

Both check the guard simultaneously — both pass — both deduct → negative balance! The product graph exposes this interleaving.
⭐ Exam topics for FM6: Drawing the product program graph for two parallel processes, tracing all interleavings, identifying race conditions, checking a safety property holds for all interleavings.
FM7 — Model Checking

Model Checking & Temporal Logic

Model checking = automatically verifying that a system satisfies a temporal property. A tool exhaustively explores all behaviours and either confirms the property holds, or gives a concrete counterexample. Target: reactive systems that run continuously (protocols, embedded controllers, concurrent programs).

FM7 — LTL & Büchi Automata

Linear Temporal Logic & Model Checking Algorithm

LTL Operators ⭐

Plain English

LTL reasons about traces — infinite sequences of states. It adds temporal operators to ordinary logic so you can say things like "eventually", "always", and "until".

OpRead asMeaningExample
X φneXt φφ holds in the very next stateX(x=0)
F φFinally φφ holds at some future point (eventually)F(done)
G φGlobally φφ holds at every future point (always)G(safe)
φ U ψφ Until ψφ holds continuously until ψ becomes true (ψ must eventually hold)active U done

LTL Property Patterns ⭐ TRANSLATE THESE IN EXAMS

EnglishLTL Formula
"The system is always safe"G(safe)
"The system never deadlocks"G(¬deadlock)
"A request is eventually granted"F(granted)
"Every request is eventually granted"G(request → F granted)
"Every dollar gets a coffee"G(dollar → F coffee)
"p holds infinitely often"G(F p)
"From some point on, p always holds"F(G p)
"Between any two a's there is a b"G(a → X(¬a U b))

LTL Semantics

A trace w = w₀ w₁ w₂ ... satisfies φ at position i (written w,i ⊨ φ):

• w,i ⊨ Xφ   iff   w,i+1 ⊨ φ
• w,i ⊨ Fφ   iff   ∃j ≥ i: w,j ⊨ φ
• w,i ⊨ Gφ   iff   ∀j ≥ i: w,j ⊨ φ
• w,i ⊨ φUψ   iff   ∃j ≥ i: w,j ⊨ ψ   and   ∀k∈[i,j): w,k ⊨ φ

Büchi Automata ⭐

Plain English

A Büchi automaton is like a DFA, but for infinite words. It "accepts" an infinite trace if the run visits an accepting state infinitely often — rather than ending in one.

DBA: A = (Q, Σ, δ, q₀, F) — same structure as a DFA.
Acceptance: An infinite word w is accepted if the run visits F infinitely often.
DFA (finite words)Büchi Automaton (infinite words)
StructureQ, Σ, δ, q₀, FSame: Q, Σ, δ, q₀, F
AcceptanceRun ends in FRun visits F infinitely often
ModelsTerminating programsReactive / non-terminating systems

LTL Model Checking Algorithm ⭐

  • System: Express as a transition system TS with states, transitions, proposition labels.
  • Property: Express as LTL formula φ.
  • Negate: Form ¬φ. A violation of φ is a witness for ¬φ.
  • Translate: Convert ¬φ to a Büchi automaton A_¬φ (tools like SPOT do this).
  • Product: Build TS ⊗ A_¬φ.
  • Emptiness check: Is there an accepting run?
      • No → φ holds for all behaviours ✓
      • Yes → counterexample found: a concrete violating trace ✗

Counterexamples: The Lasso ⭐

For LTL, a counterexample is always a lasso: a finite prefix followed by a loop that repeats forever.

s₀ → s₁ → s₂ → s₃ ↑ ↓ s₅ ← s₄ Prefix: s₀ → s₁ → s₂ Loop: s₃ → s₄ → s₅ → s₃ (repeats forever)

A lasso is: (1) finite to describe, (2) proof that the property is violated on at least one execution, (3) something you can trace manually to understand why.

CTL vs LTL — Brief Comparison

LogicQuantifies overPath quantifiersKey formulas
LTLIndividual traces (linear time)Implicit — all tracesG, F, X, U
CTLComputation tree (branching time)Explicit: A (all paths), E (exists path)AG, EF, AF, EG
CTL formulaMeaning
EF pThere exists a path where p eventually holds (reachability)
AG pOn all paths, p always holds (invariant)

CTL vs LTL incomparability: G(F p) ("p holds infinitely often on every trace") is an LTL property not expressible in CTL. AG(EF p) is a CTL property not expressible in LTL. They are different logics — neither subsumes the other.

Exam Patterns 2018–2025 ⭐

FM1–FM2: Program Graphs (appears every year)
  • Draw the program graph for a given GCL program.
  • Trace execution step-by-step from a given initial memory, writing each configuration.

  • Prove a Hoare triple {P} C {Q} using the proof rules — show each rule application explicitly.
  • Find a loop invariant for a given loop program.
  • Read or complete a proof outline (annotated program with predicates between statements).

  • Apply the consequence rule to adapt pre/postconditions.

  • Determine if one variable interferes with another — find a concrete counterexample pair (σ₁, σ₂).

  • Identify implicit information flows in conditional branches.
  • Set up the two-copy Hoare triple for a non-interference proof.

  • Translate English properties into LTL formulas using G, F, X, U.
  • Check whether a given transition system satisfies an LTL formula — give a witness or counterexample lasso.
  • Describe the Büchi acceptance condition: visits F infinitely often.
  • Note: 2024 exam used LTL over finite traces — check the exact semantics given in the exam statement.

Summary

Key Takeaways

1
The program graph is the foundation of everything. Nodes = control locations; edges = actions; configurations = (node, memory) pairs; execution = firing edges. Every other concept — operational semantics, Hoare logic, security, model checking — builds on this model. If you can draw a program graph and trace its execution, you understand the course's core.
2
Hoare triples {P} C {Q} are the language of correctness. A triple is a contract. The five proof rules (skip, assign, seq, cond, cons) let you verify contracts structurally — one construct at a time. The assignment rule's backward substitution Q[a/x] is tested in nearly every exam.
3
Loop invariants are the creative heart of Hoare logic. The three-condition checklist (P⊨I, {I∧b}C{I}, I∧¬b⊨Q) is mechanical once you have the invariant. Finding it is the skill. Ask: what is always true at the loop head? Usually it expresses "partial progress" — e.g., y·x! = n! for factorial.
4
Non-interference = secrets don't affect public outputs. Varying a secret variable should not change a public output. The two-copy technique encodes this as a Hoare triple. Type systems enforce it statically: High-level expressions cannot be assigned to Low-level variables, and cannot appear in guards controlling writes to Low variables (implicit flows).
5
LTL + Büchi automata = automated verification of temporal properties. G(p) = always p; F(p) = eventually p; G(F p) = infinitely often p; G(p→F q) = every p is eventually followed by q. Model checking finds violations as lassos — giving you a concrete, debuggable counterexample trace.
