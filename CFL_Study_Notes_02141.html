Context-Free Languages

CFL1 · CFL2 · CFL3  |  Grammars · Parse Trees · Ambiguity · ASTs · Parser Generators

Context-Free Languages describe nested, recursive structure — the kind you find in arithmetic expressions, programming languages, HTML, XML, and JSON. You will learn to define them with grammars, generate strings via derivations, visualise structure with parse trees, tame ambiguity through precedence and associativity rules, and see how the whole pipeline leads to real-world parser generators.

§ 1

What is a Context-Free Language?

Regular expressions describe flat patterns — think of matching a phone number or an email address. But some structures are nested: expressions contain sub-expressions, HTML tags wrap other tags, function calls can contain other function calls. Regular expressions cannot handle this. Context-Free Languages can.

Intuition
A language is context-free if it can be generated by rules where you can always expand a placeholder symbol regardless of what surrounds it — the rules never need to "look at the context."

Real-world examples of CFLs

  • Arithmetic expressions: 1 + 2 × (3 + 4)
  • HTML / XML documents (nested tags)
  • Programming language syntax (if/while/for)
  • JSON data structures
  • Balanced parentheses: ((()))
  • Equal-length strings: 0ⁿ1ⁿ

What CFLs cannot express

  • aⁿbⁿcⁿ — three equal counts simultaneously
  • Variable declaration before use (needs full context)
  • Type checking
  • Matching two independent counters

These require context-sensitive or Turing-complete formalisms.

§ 2

Context-Free Grammars — The Formal Definition

⭐ Exam Core

A context-free grammar is a recipe book. You start with a special placeholder symbol and keep substituting symbols according to rules until only real characters remain. There are two kinds of symbols: non-terminals (placeholders that still need expanding) and terminals (the actual characters in the final string).

Formal Definition — G = (V, T, P, S)
A CFG is a 4-tuple where each component has a precise role:
ComponentNameMeaningExample
VNon-terminalsFinite set of placeholder symbolsS, E, A, B
TTerminalsFinite set of actual characters (disjoint from V)0, 1, +, ×
PProductionsFinite set of rules A → α where A∈V, α∈(V∪T)*S → 0S1
SStart symbolThe non-terminal where every derivation begins (S∈V)S

When a non-terminal has multiple alternatives, they are written with |:

A → α₁ | α₂ | α₃ // means A can be replaced by any of α₁, α₂, α₃

The empty string (zero characters) is written ε.

⭐ Exam tip — Is it context-free?
A grammar is context-free if and only if every production has exactly one non-terminal on the left-hand side and nothing else. The rule HHF → HF is NOT context-free because the left side has two non-terminals (HH). This exact test appeared in the 2025 exam (Q1c).

Worked Example — Language 0ⁿ1ⁿ

Goal: describe {01, 0011, 000111, …} — equal numbers of 0s then 1s (n ≥ 1).

S → 01 | 0S1 // S → 01 : base case — one 0 and one 1 // S → 0S1 : wrap any valid string with another 0…1 pair

Reading: "A valid string is either 01, or a 0, then a valid string, then a 1."

§ 3

Derivations — How a Grammar Generates Strings

⭐ Exam Core

A derivation is the step-by-step process of turning the start symbol into a final string by repeatedly applying production rules. Each step replaces one non-terminal with the right-hand side of a chosen rule.

⟹One derivation step (apply one rule)
⟹*Zero or more derivation steps
Language of a Grammar
L(G) = { w ∈ T* | S ⟹* w }

The set of all terminal strings the start symbol can eventually produce.

Leftmost Derivation

At each step, expand the leftmost non-terminal. Both styles produce the same language but different step sequences.

Rightmost Derivation

At each step, expand the rightmost non-terminal. Corresponds to a different traversal order of the same parse tree.

▶ Walkthrough — Deriving 000111

Grammar: S → 01 | 0S1  ·  Target: 000111

Step 0SStart symbol
Step 10S1Apply S → 0S1
Step 200S11Apply S → 0S1 again
Step 3000111Apply S → 01 (base case) ✓
▶ Walkthrough — Deriving 0011 (two non-terminals)

Grammar: S → AB   A → 0 | 0A   B → 1 | 1B  ·  Target: 0011

Step 0SStart
Step 1ABApply S → AB
Step 20ABApply A → 0A (leftmost non-terminal)
Step 300BApply A → 0
Step 4001BApply B → 1B
Step 50011Apply B → 1 ✓
§ 4

Parse Trees — Visualising a Derivation

⭐ Exam Core

A parse tree is a picture that shows the whole derivation structure at once rather than step-by-step. It reveals the meaning of a string — which sub-expression belongs where, where an operator applies.

Rules of a Parse Tree

  • Root = the start symbol S
  • Internal nodes = non-terminals
  • Leaves = terminals (actual characters) or ε
  • Each internal node + its children = exactly one production rule
  • Reading the leaves left-to-right gives the derived string
▶ Walkthrough — Parse Tree for 0011

Grammar: S → AB   A → 0 | 0A   B → 1 | 1B

S / \ A B / \ / \ 0 A 1 B | | 0 1

Reading leaves left-to-right: 0, 0, 1, 1 → 0011 ✓

Each node's children match one production: S→AB, A→0A, A→0, B→1B, B→1.

▶ Walkthrough — Parse Tree for 1+1 with E+E grammar

Grammar: E → N | E+E | E×E | (E)   N → 0 | 1 | N0 | N1

E / | \ E + E | | N N | | 1 1

The string 1+1 has a unique parse tree here. But for 1+1×1 there are two different trees — revealing ambiguity (see §6).

§ 5

Grammars and Data — HTML, XML, JSON

One of the most practical uses of CFGs is describing data formats. HTML, XML, and JSON all have nested structure — exactly what CFGs are designed for. Understanding this connection also shows up in the mandatory assignment.

HTML Grammar (fragment)

Document → ε | Element Document Element → Text | <p> Document </p> | <ol> List </ol> List → ε | <li> Document </li> List

XML Document Type Definitions (DTDs)

A DTD is essentially a grammar written in XML's own notation. Each <!ELEMENT> defines a production — the tag name is the non-terminal; the content model is a regular expression over other non-terminals.

// DTD for email messages: <!DOCTYPE email [ <!ELEMENT email (from, to+, bcc*, subject, body)> <!ELEMENT from (#PCDATA)> // #PCDATA = plain text (terminal) <!ELEMENT to (#PCDATA)> <!ELEMENT bcc (#PCDATA)> <!ELEMENT subject (#PCDATA)> <!ELEMENT body (#PCDATA)> ]>

In CFG notation: EMAIL → <email> FROM TOP BCCN SUBJECT BODY </email>  ·  TOP → TO | TO TOP  ·  BCCN → ε | BCC BCCN

JSON Grammar

Obj → {} | {Pairs} Pairs → Pair | Pair,Pairs Pair → String : Value Array → [] | [Values] Values → Value | Value,Values Value → String | Number | Obj | Array | true | false | null
§ 6

Ambiguity

⭐ Frequently Tested

A grammar is ambiguous if there exists at least one string that can be parsed in two or more different ways — it has two or more distinct parse trees. This matters enormously: an ambiguous grammar means a compiler doesn't know which interpretation of the input is correct.

Formal Definition
A CFG G is ambiguous if there exists a string w ∈ L(G) that has more than one distinct parse tree (equivalently, more than one distinct leftmost derivation).
▶ Walkthrough — Proving 1+1×1 is Ambiguous

Grammar: E → N | E+E | E×E | (E)   N → 0 | 1 | N0 | N1

The string 1+1×1 has two distinct parse trees:

✗ Tree 1 — (+) at root → (1+1)×1 = 2
E / | \ E × E / | \ | E + E N | | | N N 1 | | 1 1
Tree 2 — (×) at root → 1+(1×1) = 2
E / | \ E + E | / | \ N E × E | | | 1 N N | | 1 1

Two different trees, two different meanings. Both produce the same numeric result here by coincidence, but with subtraction: (4−2)−2 = 0 ≠ 4−(2−2) = 4. Ambiguity causes real bugs.

⭐ Exam Pattern — Proving Ambiguity

To prove a grammar is ambiguous: exhibit two distinct leftmost derivations (or two parse trees) for the same string. The 2024 exam asked to prove a grammar ambiguous and then fix it; the 2025 exam asked to identify which productions caused ambiguities. Always give a concrete string as the example.

Where Does Ambiguity Come From?

  • Missing precedence: E → E+E | E×E doesn't say whether + or × binds tighter
  • Missing associativity: E → E+E doesn't say whether a+b+c groups left or right
  • Rule: Always try to make your grammar unambiguous!
§ 7

Operator Precedence in Grammars

⭐ Frequently Tested

Precedence determines which operator "binds tighter." In maths, × has higher precedence than +, so 1 + 2 × 3 means 1 + (2 × 3) = 7, not (1 + 2) × 3 = 9. An ambiguous grammar doesn't encode this. The fix is to stratify the grammar into separate precedence levels.

The Stratification Trick
Introduce one non-terminal per precedence level. Higher-precedence operators sit lower in the grammar hierarchy (closer to the leaves of the parse tree). Operators with lower precedence sit at the top levels.

Before — Ambiguous

E → N | E + E | E × E | (E) N → 0 | 1 | N0 | N1

+ and × have the same "status" → ambiguous

After — Unambiguous

E → T | E + T // + is low prec. T → F | T × F // × is higher F → N | (E) // atom / brackets N → 0 | 1 | N0 | N1

× sits lower (T level) → binds tighter → unambiguous

▶ Walkthrough — Parsing 1 + 1 × 1 (leftmost derivation)

With the unambiguous grammar above:

EE + TApply E → E+T
T + TApply E → T (leftmost E)
F + TApply T → F
N + TApply F → N
1 + TApply N → 1
1 + T × FApply T → T×F
1 + F × FApply T → F
1 + 1 × FApply F → N, N → 1
1 + 1 × 1Apply F → N, N → 1 ✓

The resulting parse tree forces 1 + (1 × 1) — only one tree is possible. ✓

§ 8

Associativity and Left / Right Recursion

⭐ Frequently Tested

Associativity determines how to group repeated uses of the same operator. Most operators in programming languages are left-associative. This matters critically for non-associative operators:

(4 − 2) − 2 = 0  ≠  4 − (2 − 2) = 4

Left-Associative (most operators)

2 − 1 − 1 → (2−1)−1 = 0

Enforced by left recursion — the non-terminal appears on the left of the operator:

E → E + N | N

The right side of + is N (atom), so additions can only chain to the left.

Right-Associative (e.g., exponentiation)

2 ^ 3 ^ 4 → 2 ^ (3 ^ 4)

Enforced by right recursion — the non-terminal appears on the right:

E → N + E | N

Only use this when right-associativity is the intended meaning.

Three Methods to Enforce Left-Associativity

1
Left recursion — expand the right-hand side
Change E → E+E | N to E → E+N | N. The right operand of + is forced to be an atom N, so addition can only chain leftward.
2
Right recursion (for right-associativity)
Change to E → N+E | N. Only use when you actually want right grouping.
3
Auxiliary "rest-of-list" non-terminal
An expression = a term followed by zero or more +term suffixes:
E → E' Rest E' → N | (E) Rest → ε | + E' Rest
Clean and right-recursive — avoids problems with some parsers.
⭐ Left Recursion vs. Parsing Algorithms
Left-recursive grammars are natural and correct, but naive top-down/recursive-descent parsers loop forever on left recursion. Modern parser generators handle this automatically when you declare associativity separately. This is why the course introduces parser generators — see §11.
§ 9

Abstract Syntax Trees (ASTs)

After parsing, the full parse tree contains structural "noise" — parentheses, keywords, and non-terminals that were only there to resolve ambiguity. We want only the semantically meaningful information. That's the AST.

Concrete Syntax

The original string + full parse tree. Contains all syntactic details including punctuation and structural non-terminals.

Abstract Syntax (AST)

A cleaned-up tree. Internal nodes = operations. Leaves = values. Parentheses, commas, structural tokens discarded. The tree structure encodes precedence and associativity.

▶ Walkthrough — Concrete → Abstract for 1 + 1 × 1

Concrete parse tree:

E / \ \ E + T | /|\ T T × F | | | F F 1 | | N N | | 1 1

Abstract syntax tree:

+ / \ 1 × / \ 1 1

E, T, F non-terminals gone. Structure alone encodes precedence: × is computed before +.

ASTs in Object-Oriented Code

Each grammar rule typically maps to a class. For our arithmetic grammar:

// One abstract class per grammar non-terminal public abstract class ArithExpr { } public class NumExpr extends ArithExpr { int value; } public class PlusExpr extends ArithExpr { ArithExpr left, right; } public class MultExpr extends ArithExpr { ArithExpr left, right; } // 1 + 1 × 1 in abstract syntax: ArithExpr e = new PlusExpr( new NumExpr(1), new MultExpr(new NumExpr(1), new NumExpr(1)) );

This object structure is exactly what the mandatory assignment's parser must produce — subsequent analysis stages (interpreters, verifiers, code generators) all consume this AST.

§ 10

Lexing: Characters → Tokens

Before the parser sees anything, a lexer groups the raw character stream into meaningful units called tokens. Whitespace is typically discarded. This is the first stage of the compiler pipeline.

▶ Walkthrough — Lexing a program fragment

Input characters:

if (x == 42) { y = "42 is the answer"; }

Output token stream:

IF LPAREN ID("x") EQ NUM(42) RPAREN LBRACE ID("y") ASSIGN STRING("42 is the answer") SEMICOLON RBRACE

Each token has a type (like IF, ID, NUM) and optionally a value (like the string "x" or number 42). Whitespace was discarded.

Tokens Are Defined With Regular Expressions

TokenRegular ExpressionNotes
NUM[0-9]+One or more digits
ID[A-Za-z][A-Za-z0-9]*Letter followed by letters/digits
IFifKeyword — matched literally
WHITESPACE[ \t\r\n]+Matched and discarded

The * and + operators are interpreted greedily: the lexer always takes the longest possible match. The lexer exposes a lex() function that returns the next token; the parser calls this repeatedly.

The Full Pipeline
Source text (characters)
    ↓   LEXER  — uses regular expressions
Token stream
    ↓   PARSER  — uses context-free grammar
Parse tree / AST
    ↓   ANALYSIS  — interpreter, verifier, code generator
§ 11

Parser Generators

Writing a parser by hand is difficult: simple algorithms only work on restricted grammars, and more powerful algorithms are complex and error-prone. A parser generator takes your grammar as input and automatically produces a working parser program.

What a Parser Generator Does
Your grammar file (CFG rules + precedence declarations)
    ↓   PARSER GENERATOR  (e.g., ANTLR, Happy, Yacc)
A parser program (Java / Haskell / C / ...)
    ↓   given input text
Parse tree / AST

What You Declare

  • Grammar productions in BNF-like notation
  • Operator precedence levels
  • Operator associativity (left/right)
  • Semantic actions — code to run when a rule matches (builds AST nodes)
  • Token types for the lexer

What the Tool Handles

  • Converting your grammar to a parsing algorithm
  • Resolving left recursion issues
  • Conflict detection (shift/reduce conflicts)
  • Efficient lookup tables
  • Error messages for malformed input

Mandatory Assignment Framework

The course's mandatory assignment builds a real compiler toolchain. The parser is the entry point that all later stages depend on:

Program text → [Parser = lexer + CFG] → AST → [Compiler] → Program Graph → [Interpreter / Verifier / Analyser] → Results

PA01 builds the parser and pretty-printer. PA02–PA08 build analysis tools (interpreter, security analyser, model checker, sign analyser) that all consume the AST your parser produces. This is why mastering CFGs is foundational for the whole course.

§ 12

Key Takeaways

1
A CFG = four ingredients: V, T, P, S. Every production has exactly one non-terminal on the left-hand side — that's what makes it context-free. The start symbol S kicks off every derivation. The language L(G) is all terminal strings reachable from S.
2
Derivations generate strings; parse trees picture derivations. The leaves of a parse tree read left-to-right spell out the derived string. Every internal node + its children corresponds to exactly one production rule. Both leftmost and rightmost derivations produce the same language.
3
Ambiguity = two distinct parse trees for one string. Prove it by exhibiting two different leftmost derivations. Fix it with grammar stratification (separate non-terminals per precedence level: E → T → F) and left/right recursion for associativity. Always try to make grammars unambiguous.
4
Left recursion enforces left-associativity; right recursion enforces right-associativity. E → E+N | N forces (a+b)+c. E → N+E | N forces a+(b+c). Most arithmetic operators are left-associative. Left recursion can cause infinite loops in naive parsers — use parser generators to handle it cleanly.
5
The real-world pipeline is: text → lexer (regex) → tokens → parser (CFG) → AST. The AST strips all structural noise and keeps only semantic content. Parser generators automate building the parser and let you declare precedence/associativity directly. This pipeline is the backbone of the entire mandatory assignment.
⭐ Most Common Exam Question Types (2018–2025)
  • Write a CFG for a given language — know the patterns for 0ⁿ1ⁿ, palindromes, lists, nested structures.
  • Give a derivation of a specific string — show each ⟹ step, state leftmost/rightmost.
  • Draw a parse tree — root = start symbol, leaves = terminals, reading leaves gives the string.
  • Prove ambiguity — show two distinct leftmost derivations or parse trees for the same string.
  • Fix ambiguity — stratify grammar into E/T/F levels; use left recursion for left-associativity.
  • Is this grammar context-free? — check that every production has exactly one non-terminal on the left.
