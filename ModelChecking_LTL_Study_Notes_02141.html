Model Checking & Linear Temporal Logic

Lecture notes with worked examples and exam-priority flags

📋 Summary

Model checking is an automated technique for verifying that all behaviours of a system satisfy a property. In this lecture we pair a Transition System (TS) — a graph of states and transitions — with an LTL formula φ that expresses what should be true about every run. LTL gives us operators to talk about time: what holds next, eventually, always, or until something else. These notes cover the full LTL syntax, how to write safety and liveness properties, how model checking works in practice, and the grammar rules you need for exam parse-tree questions. Past exams (2018–2025) test LTL writing and trace-checking every year — those sections are flagged ⭐.

01

What Is Model Checking? ⭐

  • Goal: Automatically verify that every possible behaviour of a system satisfies a given property — no manual proof needed.
  • Input: A Transition System (TS) and a formula φ.
  • Output: TS ⊧ φ (yes/no) — and if no, a counterexample trace.
  • Unlike testing: Model checking covers all executions, not just sampled ones.

The Transition System (TS)

A TS is a formal model of a system's behaviour. It has five components:

ComponentNotationMeaning
StatesSAll possible configurations of the system
Initial statesI ⊆ SStates where execution can start
Transition relation→ ⊆ S × SWhich states can follow which
Atomic propositionsAPBasic yes/no facts about states (e.g. crit = 1)
Labelling functionL : S → 2APWhich APs are true in each state

Traces

  • A path in the TS is an infinite sequence of states: s₀ → s₁ → s₂ → …
  • A trace is the corresponding sequence of label-sets: L(s₀) L(s₁) L(s₂) …
  • Traces(TS) = the set of all traces from all initial states.
  • TS ⊧LTL φ means: every trace of TS satisfies φ.
💡
Plain English: Think of the TS as a graph where nodes are "snapshots" of the system and edges are "what can happen next." A trace is an infinite walk through this graph, recording what was true at each step. Model checking asks: "Is the property φ true on every such walk?"
02

LTL Syntax ⭐

An LTL formula is built from these ingredients:

  • Atomic propositions: p, q, r, … — basic facts that are true or false at a given state (e.g. crit=1, x<0).
  • Truth constant: tt (always true), ff (always false).
  • Boolean connectives: ¬φ, φ₁ ∧ φ₂, φ₁ ∨ φ₂, φ₁ → φ₂
  • Temporal operators: Xφ, Fφ, Gφ, φ₁ U φ₂

The Four Temporal Operators

Xφ
Next
φ holds in the very next state on the trace.
π, i ⊧ Xφ iff π, i+1 ⊧ φ
Fφ
Finally / Eventually
φ holds at some future state (possibly now).
π, i ⊧ Fφ iff ∃j ≥ i. π, j ⊧ φ
Gφ
Globally / Always
φ holds at every state from now on — forever.
π, i ⊧ Gφ iff ∀j ≥ i. π, j ⊧ φ
φ₁ U φ₂
Until
φ₁ holds continuously until φ₂ eventually holds (φ₂ must come).
∃j ≥ i. π,j ⊧ φ₂ ∧ ∀k∈[i,j). π,k ⊧ φ₁
⚠️
Common mistake: With U, φ₂ must eventually hold. You can't just have φ₁ forever — that would require Gφ₁. Also don't confuse F ("at least once") with G ("every time").

Derived Operators

  • Fφ = tt U φ  —  "true until φ" = φ eventually holds.
  • Gφ = ¬F¬φ  —  "it never eventually becomes not-φ" = φ always holds.

These mean you only strictly need U and ¬ — F and G are shorthand.

03

Writing Properties in LTL ⭐

Most properties fall into a small number of patterns. Recognising the pattern is the fastest route to the right formula.

PatternNatural LanguageLTL Formula
Safety"Bad thing never happens"G ¬bad
Liveness"Good thing eventually happens"F good
Repeated liveness"Good thing happens infinitely often"GF good
Response"Whenever request, eventually response"G(request → F response)
Mutual exclusion"Two processes never both in crit section"G ¬(crit₁ ∧ crit₂)
Progress"Wanting to enter implies eventually entering"G(want → F crit)

How to Read an LTL Formula Aloud

  • Start from the outermost operator and work inward.
  • G(p → Fq) → "Always: if p holds, then eventually q holds."
  • GF p → "Always eventually p" = "p happens infinitely often."
  • FG p → "Eventually always p" = "p is true from some point on forever."
  • G ¬(a ∧ b) → "It is never the case that both a and b are true at the same time."
🗣
Translation trick: Replace G with "always", F with "eventually", X with "in the next step", and U with "until". Then read left-to-right. Practice this until it's automatic — exam questions often ask you to translate in both directions.
04

How Model Checking Works ⭐

  • To verify TS ⊧ φ: check that ALL traces satisfy φ.
  • To find a counterexample (a trace that violates φ): check TS ⊧ ¬φ. If this holds, then some trace satisfies ¬φ — i.e., violates φ.
  • Counterexamples in LTL always have the shape of a lasso: a finite prefix followed by an infinite loop.

Witnesses and Counterexamples

  • A witness shows why a formula holds. For existential properties it's a single trace; for universal (G) properties it's the whole TS.
  • A counterexample shows why a formula fails — just one violating lasso suffices.
  • LTL model checking has explainability built in: the tool always gives you a counterexample when it fails.

The State Explosion Problem

  • The TS can be exponentially large in the number of concurrent processes or variables.
  • n boolean variables → up to 2ⁿ states.
  • This is the main practical limitation. Techniques like symbolic model checking, abstraction, and partial-order reduction help manage it.
05

LTL Grammar ⭐

The exam often asks you to draw a parse tree for an LTL formula using the course grammar and resolve ambiguity using precedence rules.

The Course Grammar

F → tt | A | !F | F & F | XF | GF | F U F | [F]
  • A stands for any atomic proposition.
  • [F] means parentheses (explicit grouping).
  • This grammar is ambiguous — the same string can have two parse trees.

Precedence Rules (highest binds tightest → lowest)

PriorityOperatorMnemonic
1 (tightest)! (negation)Unary, always binds first
2& (and)Like multiplication
3X (next)
4G (always)
5 (loosest)U (until)Binds last without brackets

Both & and U are left-associative: a U b U c parses as (a U b) U c.

⚠️
Exam tip: When drawing a parse tree, first identify the loosest (lowest-priority) operator that isn't inside brackets — that becomes the root. Then recurse on each sub-formula. Always check associativity when the same operator appears twice.

Worked Examples

EXAMPLE 1 Reading LTL — Translating and checking a formula on a trace

Task: Given the formula G(p → F q) and the trace π = {p} · {p} · {q} · {p} · {q} · {q} · … (assume it repeats), determine if π satisfies the formula.

1
Parse the formula — identify the outermost operator.
The outermost is G, applied to (p → F q).
Inside: an implication with left-hand side p and right-hand side F q.
2
Translate to English.
G(p → F q) = "Always: if p holds, then eventually q holds."
In other words: every time you see p in the trace, q must appear at some point afterwards (or now).
3
Check the trace position by position.
We only need to check positions where p holds (the implication is vacuously true otherwise).
✅
i = 0: label = {p}. p holds here. Is there a future j with q ∈ L(sⱼ)? Yes — j=2 has label {q}. ✓
✅
i = 1: label = {p}. p holds. Next q is at i=2. ✓
⬜
i = 2: label = {q}. p does not hold. Implication is vacuously satisfied. ✓
✅
i = 3: label = {p}. p holds. Next q is at i=4. ✓
The pattern repeats: every p is followed by a q. The formula holds on this trace. ✓
EXAMPLE 2 Writing LTL — "x cannot be negative for more than two states in a row"

Task: Express the constraint using only the atomic proposition x<0.

1
Think about what the violation looks like.
A violation is: three consecutive states where x<0 is true.
In LTL terms: at some position i, we have x<0 at i, i+1, and i+2.
2
Build the formula to rule out the violation.
"If x<0 now AND x<0 in the next state, then x<0 must be false two steps from now."
G ( (x<0 ∧ X x<0) → XX ¬x<0 )
3
Verify the formula captures the right meaning.
  • The G says: check this at every position.
  • The antecedent x<0 ∧ X x<0 is true exactly when we've seen two consecutive negatives.
  • The consequent XX ¬x<0 forces the third step to be non-negative.
  • Result: you can have at most 2 consecutive negative states.
EXAMPLE 3 Mutual Exclusion — Writing safety and liveness, then checking a trace

Scenario: Two processes compete for a critical section. We have atomic propositions crit₀, crit₁ (in critical section), and want₀, want₁ (wanting to enter).

1
Identify the atomic propositions.
  • crit₀ — process 0 is in its critical section
  • crit₁ — process 1 is in its critical section
  • want₀ — process 0 wants to enter
  • want₁ — process 1 wants to enter
2
Write the Safety formula (mutual exclusion = "never both in crit"):
G ¬(crit₀ ∧ crit₁)
Plain English: "Always, it is not the case that both processes are simultaneously in their critical section."
3
Write the Liveness formula (progress = "wanting implies eventually entering"):
G ( (want₀ ∨ want₁) → F(crit₀ ∨ crit₁) )
Plain English: "Always, if any process wants to enter, then eventually some process does enter."
4
Check a candidate trace against the Safety formula.
Suppose the trace is:
s₀: {want₀} → s₁: {want₀, want₁} → s₂: {crit₀, want₁} → s₃: {want₁} → s₄: {crit₁} → s₅: {} → …
✅
s₀: {want₀} — neither crit. Safety holds. ✓
✅
s₁: {want₀, want₁} — still no crit. Safety holds. ✓
✅
s₂: {crit₀, want₁} — only crit₀. ¬(crit₀ ∧ crit₁) = ¬(true ∧ false) = true. ✓
✅
s₃–s₄: Only one process in crit at a time. Safety holds throughout. ✓
Now check liveness: at s₁ both processes want to enter. Does one eventually enter? Yes — s₂ has crit₀. At s₂, process 1 still wants: does it eventually enter? Yes — s₄ has crit₁. Both properties hold on this trace. ✓
EXAMPLE 4 ⭐ Grammar — Drawing the parse tree for p U q & !r

Task: Parse p U q & !r using the course grammar and precedence rules.

1
Apply precedence rules to find the root operator.
Operators present: U (priority 5), & (priority 2), ! (priority 1).
The loosest operator (highest priority number = binds last) is U (priority 5). So U is the root.
2
Split into left and right of U.
Left of U: p
Right of U: q & !r
3
Parse the right sub-formula q & !r.
Operators: & (priority 2), ! (priority 1).
Loosest is & → root of this sub-tree.
Left: q, Right: !r.
Then ! applies to r.
4
Final parse tree:
U / \ p & / \ q ! \ r
So the formula means: "p holds until (q AND NOT r)".
07

Exam-Style Tips ⭐

Parsing an LTL Formula

  • Find the operator with the lowest priority (not inside brackets) — that's the root.
  • When in doubt: U is almost always the root if present at the top level.
  • Use brackets […] to override default precedence.
  • Remember left-associativity: a U b U c = (a U b) U c.

Writing LTL for Natural-Language Properties

  • "Never X" → G ¬X
  • "Eventually X" → F X
  • "X always leads to Y" → G(X → F Y)
  • "X infinitely often" → GF X
  • "After some point, always X" → FG X
  • "X until Y (and Y must come)" → X U Y

Common Mistakes to Avoid

  • Mixing up F and G: F = "at least once", G = "every single time".
  • Forgetting U requires φ₂ to eventually hold: p U q means q must come eventually. If q never appears, the formula is violated even if p holds forever.
  • Confusing GF and FG: GF p = p infinitely often; FG p = p true from some point on forever. These are very different!
  • Wrong root in parse tree: Always use precedence, not left-to-right reading, to find the root.
  • Forgetting to check all initial states: TS ⊧ φ requires every initial state's trace to satisfy φ.

🎯 Key Takeaways

1
Model checking = checking ALL traces. ⭐
TS ⊧LTL φ means every infinite trace from every initial state satisfies φ. A single counterexample (a lasso) is enough to refute it. The counterexample is always produced by the tool.
2
Know all four operators cold. ⭐
Xφ = next state; Fφ = some future; Gφ = every future; φ U ψ = φ holds until ψ inevitably arrives. Confusing F and G is the most common exam error.
3
Master the standard property patterns. ⭐
Safety = G¬bad; liveness = GF good; response = G(req → F resp); mutual exclusion = G¬(crit₀ ∧ crit₁). These templates appear on almost every exam.
4
Precedence rules resolve grammar ambiguity. ⭐
Order (tightest to loosest): ! → & → X → G → U. The loosest operator not in brackets is always the root of the parse tree. Both & and U are left-associative.
5
Checking a trace manually: work position by position. ⭐
For G(p → Fq), scan every position: whenever p is true, scan forward to confirm q appears. For U, confirm φ₂ does arrive. For F/G, check all (G) or any (F) future positions as needed.
