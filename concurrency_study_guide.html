Section 1

What is Concurrency?

So far in the course you have studied programs that run sequentially — one step at a time, one process at a time. Concurrency changes that: multiple processes run simultaneously, potentially reading and writing the same shared data, or communicating with each other over channels.

Definition
Concurrency = a system consisting of several processes that execute in parallel and interact by exchanging information — either through shared memory or over channels.

Why Concurrency Is Hard

  • The order in which the individual steps of different processes execute is non-deterministic — we cannot predict it in advance.
  • This creates an enormous space of possible interleavings: if two processes each have n steps, there can be exponentially many orderings.
  • A program might work correctly in 99.9% of interleavings, but break in one rare ordering — making bugs extremely hard to reproduce or test.
  • Classic problems include race conditions (two processes clobber each other's data) and deadlock (processes wait for each other forever).

The Two Main Models ⭐

Shared Variables

  • Processes communicate by reading/writing a common memory.
  • Simple to program but dangerous — no built-in coordination.
  • Used in the par/rap construct of GCL.

Channel-Based Communication

  • Processes send/receive values over named channels (c!a, c?x).
  • Can be asynchronous (buffered) or synchronous (hand-shake).
  • More structured; no shared memory needed.

Guarded Commands with Concurrency — par/rap ⭐

Syntax

The GCL language is extended with a new top-level program construct:

par
  C1
[]
  C2
[]
  ...
[]
  Cn
rap
  • par ... rap launches n concurrent processes.
  • Each Ci between [] separators is an independent process.
  • The processes share the same memory (shared variables).
  • There must be at least one process (n ≥ 1).
  • Each Ci is written in ordinary GCL (with do...od, if...fi, etc.).

Program Graphs — One Per Process ⭐

To give semantics to par C₁ [] ... [] Cₙ rap, we construct n separate, disjoint program graphs — one for each process — exactly as we did in Chapters 1–2, but making sure all node names are distinct across graphs.

  • Graph Ei has its own start node qi⊳ and end node qi◁.
  • The nodes of different graphs are kept completely separate.
  • Edges encode the same actions (skip, assignments, guards) as before.

Configurations — The New State Format ⭐

In a single-process program, a configuration was ⟨q; σ⟩ — one node and one memory. With n processes we need to track where every process is:

Configuration (Shared Variables)
⟨q₁ q₂ ⋯ qₙ ; σ⟩
  • qi is the current node of process i in its own program graph.
  • σ is the shared memory (a single memory for all processes).

Execution Steps — Interleaving Semantics ⭐

At each step, exactly one process moves. If process i has an edge (qi○, α, qi●) and action α is enabled in memory σ, then:

⟨q₁ ⋯ qi○ ⋯ qₙ ; σ⟩  ─α→  ⟨q₁ ⋯ qi● ⋯ qₙ ; σ'⟩
      only process i moves       all other q's unchanged
  • All other process nodes qj (j ≠ i) remain exactly as they were.
  • The memory σ is updated to σ' according to the semantics of α (as always).
  • This is the interleaving semantics: concurrency is modelled as non-deterministic sequential interleaving.
  • An execution sequence is a chain of such steps starting from the initial configuration ⟨q₁⊳ q₂⊳ ⋯ qₙ⊳ ; σ₀⟩.
Key Insight
At any step, the scheduler picks any enabled process to move. We do not control or predict this choice — hence the non-determinism.
Section 3

Shared Variables & Race Conditions ⭐

What is a Race Condition?

A race condition occurs when two or more processes access the same shared variable concurrently, and at least one of them writes to it. The final result depends on which process "wins the race" — i.e., on the interleaving order — which we cannot control.

Concrete Example: Alice's Sum Problem

Alice wants to sum all values in an array A. She tries to speed this up using two threads:

par
  do i <= n → sum := sum + A[i] ; i := i+1 od
[]
  do i < n → sum := sum + A[n] ; n := n-1 od
rap

Both threads read and write sum. Consider what happens when both reach their loop body at the same time, with sum = 10, A[i] = 5, A[n] = 3:

StepWho movesActionsum afterNote
1Thread 1reads sum = 1010Both read old value
2Thread 2reads sum = 1010
3Thread 1writes sum := 10+5 = 1515
4Thread 2writes sum := 10+3 = 1313⚠ Thread 1's update LOST!
  • The correct answer would be 18 (10+5+3), but we got 13 — Thread 1's addition was silently erased.
  • This happens because the read-then-write sequence is not atomic: another thread can intervene between the read and the write.
  • The problem is non-deterministic: sometimes the result is correct, sometimes not. This makes race conditions very hard to debug.
Why This Is Dangerous
In real software, race conditions can cause bank accounts to go negative, data to be corrupted, or security vulnerabilities. The key lesson: shared mutable state requires explicit coordination.
Section 4

Mutual Exclusion ⭐

The mutual exclusion problem: multiple concurrent processes need to access a shared resource (the critical section), but only one process may be in the critical section at any given time.

Safety and Liveness ⭐

⭐ Safety Property

At most one process is in the critical section at any time.

To argue: show that no reachable configuration has two processes simultaneously at their critical section nodes.

⭐ Liveness Property

If a process wants to enter the critical section, it eventually will (given no process stays in permanently).

To argue: show every execution sequence eventually reaches the critical section node for any waiting process.

Lamport's Bakery Algorithm ⭐

The Intuition

Think of a bakery where customers take a numbered ticket and wait for their number to be called. The process with the lowest ticket wins. In the two-process version, each process has a shared variable x1 or x2 representing its ticket number. A ticket of 0 means "not interested."

The GCL Code

par
  do true →
    x1 := x2 + 1 ;           -- Take a ticket (one more than Process 2's)
    if x2 = 0 ∨ x1 < x2 →   -- Wait until I have the lower ticket (or P2 has none)
      critical section1      -- CRITICAL SECTION
    fi ;
    x1 := 0                  -- Give up ticket
  od
[]
  do true →
    x2 := x1 + 1 ;           -- Take a ticket
    if x1 = 0 ∨ x2 < x1 →   -- Wait until I have the lower ticket (or P1 has none)
      critical section2      -- CRITICAL SECTION
    fi ;
    x2 := 0                  -- Give up ticket
  od
rap

How it Works — Plain English

  • Taking a ticket: Process 1 sets x1 := x2 + 1. This gives it a number one higher than whatever Process 2 currently holds — a simple way to get a bigger ticket.
  • Checking priority: The guard x2 = 0 ∨ x1 < x2 reads: "I can enter the critical section if Process 2 has no ticket (x2=0) OR my ticket number is strictly lower."
  • Waiting: If neither condition holds, the guard is false, so Process 1 is stuck — it cannot take the if branch and must wait.
  • Releasing: After the critical section, x1 := 0 cancels the ticket, signalling "I am no longer interested." This unblocks Process 2.
  • Why safe? Both processes cannot both have the lower ticket: if x1 < x2 then x2 < x1 is false (and vice versa), so at most one can enter.
  • Why live? A waiting process will always eventually see the other's ticket become 0 (when it exits), freeing it to enter.

Program Graph Nodes (Process 1)

NodeMeaningEdge out / action
q₁⊳Start of outer looptrue → q₁¹
q₁¹Just executed x1 := x2+1x1 := x2+1 → q₁²
q₁²Evaluating guard for critical sectionx2=0 ∨ x1<x2 → q₁³
q₁³Inside critical sectionE₁ (critical section body) → q₁⁴
q₁⁴After critical sectionx1 := 0 → q₁◁
q₁◁End of iteration, loops back→ q₁⊳

Process 2 has the symmetric graph with x2 := x1+1 and guard x1=0 ∨ x2<x1.

Peterson's Algorithm (Mentioned in Exercises)

  • Another mutual exclusion algorithm, slightly simpler than Bakery for 2 processes.
  • Uses a turn variable as a "tiebreaker" plus wants1, wants2 flags.
  • The guard for Process 1 to enter: (turn=1) ∨ (wants2=0) — "it's my turn, or Process 2 doesn't want in."
  • The course exercises ask you to implement it in GCL and verify safety/liveness using par4fun.
Section 5

Asynchronous vs. Synchronous Communication ⭐

Beyond shared variables, GCL can be extended with channels. Two new actions are introduced:

Channel Actions
  • c!a — output: send the value of expression a on channel c.
  • c?x — input: receive a value from channel c and assign it to variable x.

Asynchronous (Buffered)

  • Each channel has a buffer (a queue of values).
  • c!a: sender immediately appends the value to the buffer. Sender does not wait.
  • c?x: receiver takes the front value from the buffer. Blocks if buffer is empty.
  • Configuration: ⟨q⃗; σ, κ⟩ where κ maps each channel to its buffer contents.
  • One process moves at a time (individual execution steps).

Synchronous (Handshake)

  • No buffer. Sender and receiver must be ready simultaneously.
  • The two actions c!a (in process i) and c?x (in process j) fuse into a single joint action c!a?x.
  • Both processes advance their nodes together in one step.
  • Configuration: ⟨q⃗; σ⟩ — no buffer needed.
  • Two types of step: individual (one process) and joint (two processes).

Synchronous Execution Step Rule ⭐

For synchronous communication, when process i wants to send c!a and process j wants to receive c?x, they take a joint step:

⟨q₁ ⋯ qi○ ⋯ qj○ ⋯ qₙ ; σ⟩  ─c!a?x→  ⟨q₁ ⋯ qi● ⋯ qj● ⋯ qₙ ; σ'⟩
  provided: (qi○, c!a, qi●) ∈ Ei  AND  (qj○, c?x, qj●) ∈ Ej
  result:   σ' = σ[x ↦ ⟦a⟧σ]  (x gets the value of a in memory σ)
  • The joint action c!a?x only appears in the execution sequence, not in any individual program graph.
  • Both processes i and j must have the matching send/receive at the right moment — if only one is ready, neither can proceed alone on that channel.

Broadcast and Gather (Bonus)

  • c!!k a: broadcast — sends a to at least k processes simultaneously.
  • c??k A x: gather — receives from at least k senders simultaneously into array A.
  • These are joint actions involving more than two processes at once.

Worked Example — Bakery Algorithm Step by Step ⭐

We trace through the Bakery Algorithm with initial memory σ₀ = [x1 ↦ 0, x2 ↦ 0, crit ↦ 0]. We use crit as a counter of processes in the critical section (useful for model checking; you don't need it for the trace itself).

1

Write the par/rap Program

par
  do true →
    x1 := x2 + 1 ;
    if x2 = 0 ∨ x1 < x2 →
      critical section1
    fi ;
    x1 := 0
  od
[]
  do true →
    x2 := x1 + 1 ;
    if x1 = 0 ∨ x2 < x1 →
      critical section2
    fi ;
    x2 := 0
  od
rap
2

Each process gets its own graph. Here are the nodes for Process 1 (Process 2 is symmetric):

q₁¹After taking ticket: x1 := x2+1
q₁²Evaluating entry guard: x2=0 ∨ x1<x2

q₁⁴After critical section, about to release ticket

Process 2 has nodes q₂⊳, q₂¹, q₂², q₂³, q₂⁴ with symmetric actions.

At no point are both P1 and P2 simultaneously at q₁³ and q₂³. Safety holds. Also, P2 eventually gets in (step 8), so liveness holds.
4

Trace a Bad Interleaving (Without a Mutex Algorithm)

Suppose we removed the ticket system and both processes just checked a simple shared flag incorrectly. Consider two processes that both do:

do true →
  if crit = 0 →         -- "Is the room free?"
    crit := 1 ;          -- "I'm going in!"
    // critical section
    crit := 0
  fi
od
StepActioncritProblem
1P1: reads crit = 0 (guard true)0
2P2: reads crit = 0 (guard true)0Before P1 writes!
3P1: writes crit := 11
4P2: writes crit := 11⚠ BOTH now in critical section!

The window between reading the guard and writing the flag allows both processes to slip through simultaneously. Safety is violated.

5

How the Bakery Algorithm Prevents the Bad Interleaving

  • Tickets are unique per round: When P1 sets x1 := x2+1, it reads the current value of x2. Even if both do this almost simultaneously, the ticket values will differ — one will always be strictly larger.
  • The guard is strict: The condition x1 < x2 uses strict inequality. It is mathematically impossible for both x1 < x2 and x2 < x1 to be true simultaneously.
  • The algorithm handles the corner case: If x2 = 0 (P2 has no ticket), P1 can freely enter. This prevents P1 from starving when P2 is not interested.
  • Releasing the ticket is atomic: Setting x1 := 0 is one indivisible step. After it, P2's guard immediately becomes true (assuming x2 > 0), so P2 can proceed.
Why It Cannot Go Wrong
For both processes to be in the critical section, we would need simultaneously: x2=0 ∨ x1<x2 (P1's guard) AND x1=0 ∨ x2<x1 (P2's guard). If both have nonzero tickets, we need x1<x2 AND x2<x1 — a contradiction. If one has ticket 0, that means it already exited and set its ticket to 0 — so it isn't in the critical section.
Section 7

Exam-Style Tips ⭐

Writing Execution Sequences

Format
Always write: ⟨nodeP1 nodeP2 ; memory⟩ ─action→ ⟨nodeP1' nodeP2' ; memory'⟩. Label which process acts. Show each step individually — do not skip steps.
  • Write the full configuration at each step: all process nodes and all relevant variable values.
  • For each step, state which process moves, what edge it takes, and how memory changes.
  • If the action is a guard (boolean), write the guard value and say whether it's true/false.
  • For channel communication: write the joint action c!a?x and which two processes participate.

Arguing Safety

Template Answer
"We must show that no reachable configuration has both P1 at node q₁³ and P2 at node q₂³ simultaneously. Suppose for contradiction both were at their critical section nodes. Then P1's guard x2=0 ∨ x1<x2 was true and P2's guard x1=0 ∨ x2<x1 was true. Since both are in the critical section, their ticket values are nonzero. So we need x1<x2 AND x2<x1 — a contradiction. Hence safety holds."

Arguing Liveness

Template Answer
"We must show that if a process is waiting (stuck at the guard), it will eventually be released. If P1 is stuck at q₁², then x2 ≠ 0 AND x1 ≥ x2. Since P2 is in the critical section with ticket x2, and no process stays in the critical section permanently, P2 will eventually execute x2 := 0. After this, P1's guard becomes true (since x2=0 satisfies x2=0 ∨ x1<x2), so P1 can enter."

Common Exam Phrasings

  • "Construct an execution sequence that demonstrates…" — write a concrete trace showing the required behaviour.
  • "Find an execution sequence that violates safety" — look for a trace where both processes reach their critical section simultaneously. Work backwards from that bad state.
  • "Argue that the algorithm guarantees the safety/liveness property" — write a short proof by contradiction (safety) or progress argument (liveness).
  • "Describe the semantics of par C1 [] C2 rap" — mention: n disjoint program graphs, configurations are tuples ⟨q₁ q₂; σ⟩, only one process moves at a time (interleaving), execution steps advance one process node while others stay fixed.
  • "What is a race condition? Give an example." — shared variable read-modify-write split by an interleaving step.

Key Formal Definitions to Know

  • ⭐ Execution step rule for shared variables: ⟨q₁⋯qᵢ○⋯qₙ; σ⟩ →α ⟨q₁⋯qᵢ●⋯qₙ; σ'⟩ whenever (qᵢ○, α, qᵢ●) ∈ Eᵢ and ⟦α⟧σ = σ'.
  • ⭐ Synchronous joint step: ⟨…qᵢ○…qⱼ○…; σ⟩ →c!a?x ⟨…qᵢ●…qⱼ●…; σ'⟩ whenever process i has send edge and process j has receive edge on same channel.
  • Asynchronous step: one process moves; buffer κ is updated (output appends to right, input removes from left).
  • Product Program Graph (Exercise 8.10): nodes Q₁×Q₂×⋯×Qₙ, same execution sequences as interleaving semantics. Size grows exponentially — this is the state explosion problem.

⭐ Key Takeaways

1
Concurrency = non-deterministic interleaving. The par/rap construct in GCL runs n processes "simultaneously." The semantics models this as all possible interleavings — one process moves at a time. This non-determinism is the root of all concurrency bugs.
2
Configurations track all process locations. A concurrent configuration is ⟨q₁ q₂ ⋯ qₙ ; σ⟩. You must write the node of every process and the full memory at every step of an execution sequence.
3
Safety vs. Liveness are distinct properties. Safety = "nothing bad ever happens" (at most one process in the critical section). Liveness = "something good eventually happens" (a waiting process eventually enters). An algorithm must satisfy both. Arguing them requires different proof strategies.
4
The Bakery Algorithm guarantees both safety and liveness through the ticket mechanism: tickets ensure a total order, so two processes cannot simultaneously have the "winning" ticket. Safety follows from strict inequality; liveness follows because tickets are always eventually released.
5
Synchronous communication requires joint actions. In synchronous channel semantics, a send c!a and a matching receive c?x fuse into one joint step c!a?x that advances both processes simultaneously. There is no buffer — this is fundamentally different from asynchronous communication where sender and receiver are decoupled via a queue.
