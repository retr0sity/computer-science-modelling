RL / CFL Task 1 – Formal Languages ⭐ ▾

Context: A DFA A over Σ = {0,1} with 6 states, and a regular expression E = 1*(01)*. Questions cover DFA minimisation, language set operations, grammar context-freeness, ambiguity, and disambiguation.

⭐ Q1a
Minimise DFA A using the table-filling method to produce an equivalent DFA A'. Label each state of A' with subsets of states from A.
▶ Show Solution
Solution

States {q0,q2} and {q3,q5} can be merged. The minimised DFA A' has 4 states: {q0,q2} (initial, accepting), {q3,q5} (accepting), q1, and q4 (trap/sink). The language is L(A) = (01 + 10)*.

⭐ Q1b
For language set operations between L(A'), L(E) (where E = 1*(01)*), fill a table with the shortest word in each language. Operations: ∩, ∪, \, and Kleene star unions.
▶ Show Solution
Solution

Key results: L(A') ∩ L(E) → ε (empty string is in both). L(A') \ L(E) → 10 (in A' but not E since E requires 1* prefix). L(E) \ L(A') → 1 (the string "1" is in E but not A'). L(A)* ∪ L(E)* → ε.

⭐ Q1c
Given a grammar G with productions for F, A, N, R including a production "HHF →", is G context-free? Find the minimal set of productions to remove to make it context-free.
▶ Show Solution
Solution

Production (6): A → HF makes G non-context-free because the left-hand side has two non-terminals (it is actually the production HHF that causes this). A context-free grammar requires exactly one non-terminal on the left-hand side. Remove production 6 only — that is the minimal fix.

⭐ Q1d
After removing the non-CF production, does the resulting grammar G' have ambiguities? Show an example if yes.
▶ Show Solution
Solution

Yes. Ambiguous production pairs: (2,2), (2,4), (2,5). Example using (2,5) — the string a ∧ Hb has two leftmost derivations:

F ⇒ F∧F ⇒ a∧F ⇒ a∧HF ⇒ a∧Hb F ⇒ F∧F ⇒ F∧HF ⇒ F∧Hb ⇒ a∧Hb

Operator precedence between ∧ and H is undefined, causing two distinct parse trees.

⭐ Q1e
Fix the ambiguity from Q1d by providing a new unambiguous grammar. Explain changes in 1-3 lines.
▶ Show Solution
Solution

Use stratification to enforce precedence. Define F0 as the lower-precedence level (∧ binds looser) and F1 as the higher-precedence level (H binds tighter):

F0 → F1 | F0 ∧ F0 F1 → A | A⊙N | OF1 | HF1

This forces H to bind tighter than ∧, eliminating the ambiguity.

FM Task 2 – Programming Language Semantics ⭐ ▾

Context: A function bin() that transforms GCL programs by flattening nested expressions to at most one binary operation per assignment. Questions probe correctness, bugs, and properties of this transformation.

⭐ Q2a
Apply bin() to: x := a + b * c - d, where all are variables.
▶ Show Solution
Solution
x := b*c; x := a+x; x := x-d

bin() splits from the inside out: first compute b*c into x, then a+x, then subtract d.

⭐ Q2b
Identify a bug in bin(). Provide memories σ1, σ2, σ3 and a program C such that C (starting at σ1) reaches σ2, but bin(C) (starting at σ1) reaches σ3 ≠ σ2.
▶ Show Solution
Solution

Bug: bin() stores intermediate values back into the variable being computed — but that variable may appear later in the same expression, corrupting it.

C: x := 0 + 1 + x bin(C): x := 0 + 1; (sets x to 1, overwriting original x) x := x + x (computes 1+1=2 instead of 1+0=1)

σ1: x=0, σ2: x=1, σ3: x=2

Q2c
A program graph is shown — it is bin(C) of some command C, where only one assignment was split. Recover C.
▶ Show Solution
Solution
r := 0; i := 0; do i < n -> i := i+1; r := i + 2*r od
Q2d
True or False: 8 statements about properties of bin() — determinism, idempotency, structural equivalence of program graphs, etc.
▶ Show Solution
Solution (one per line)
1. False — bin() bug can break evolving property 2. False — bin() bug can break determinism 3. True — bin(bin(C)) = bin(C) since split cases never re-split 4. True — e.g. bin(x:=0) = bin(x:=(0)) 5. False — e.g. x:=1+1+1 has different PG than bin of it 6. True — bin() never reduces edges 7. False — PG with multi-op assignment can't come from bin() 8. False — C and C' can differ in stuck branches
Q2e
Verify each step of Alice's theorem proof for A⟦a1@a2⟧σ = A⟦x@a2⟧σ[x↦A⟦a1⟧σ]. Mark each of the 4 steps Correct or Incorrect.
▶ Show Solution
Solution
Step 1: Correct — applies definition of A correctly Step 2: Correct — x evaluates to A⟦a1⟧σ in the updated memory Step 3: Incorrect — ignores that x may appear in a2 (the core bug!) Step 4: Correct — standard A semantics
SEC / VER Task 3 – Verification & Analysis ⭐ ▾

Context: A GCL exponentiation loop (r := e*r; i := i+1) with a power-consumption counter. All variables are secret. Questions combine information flow analysis, non-interference, Floyd-Hoare triples, and power side-channels.

⭐ Q3a
Using information flow analysis (fm4fun / inspectify), list all variables that flow into variable "power".
▶ Show Solution
Solution
power i n

Variable e does NOT flow into power — the number of loop iterations depends only on n, and power increments are fixed at +1 and +1 per iteration.

⭐ Q3b
Formalise the non-interference property "Trent cannot learn e through power" as a Floyd-Hoare triple on a duplicated program. What can Alice conclude?
▶ Show Solution
Solution
{ 0<=n1 & 0<=e1 & 0<=n2 & 0<=e2 & n1=n2 & power1=power2 } [copy 1 of program]; [copy 2 of program] { power1 = power2 }

This triple verifies in Chip. Conclusion: Trent cannot learn anything about e by observing power — power depends only on n.

⭐ Q3c
Provide a loop invariant for the annotated Floyd-Hoare triple with postcondition {r = exp(e,n) ∧ power = 2+2*n}.
▶ Show Solution
Loop Invariant
i >= 0 & i <= n & r = exp(e,i) & power = 2+2*i

This captures: i counts iterations, r holds e^i, and power tracks 2 (init) + 2 per iteration. At loop exit (i=n), gives the postcondition.

Q3d
If power is incremented by e (not 1) per multiplication (power := power + e), find the new precise postcondition for power.
▶ Show Solution
Solution
r = exp(e, n) & power = 2 + (1+e)*n

Now power depends on both e and n — so power leaks information about e!

MC / SA Task 4 – Model Checking & Sign Analysis ⭐ ▾

Context: Building transition systems from sign analysis assignments. AP = PowerSet({−,0,+}). Three candidate transition systems T S1, T S2, T S3 given. Questions test whether they can arise from real program graphs, and LTL evaluation.

⭐ Q4a
For each of TS1, TS2, TS3, determine whether there exists a program P, analysis A, and variable x such that TSi = TSP,A,x.
▶ Show Solution
Solution
TS1: NO — s2 labelled {0} can't have successors {+} and {−} (a conditional node can't change the sign) TS2: YES — construct: x:=-1; x:=x-x; if x>0->skip [] x<0->skip fi TS3: NO — the control flow graph structure is impossible
⭐ Q4b
For each LTL property φ, does TS2 ⊨ φ and TS3 ⊨ φ? Properties: X−, X+, F(¬−∧¬+), G+, GF(−∧+).
▶ Show Solution
Solution (TS2, TS3)
X− → YES, YES X+ → NO, NO F(¬−∧¬+) → NO, YES G+ → NO, NO GF(−∧+) → YES, NO
Q4c
Formalise in LTL (using only proposition x<0): (1) x cannot be negative more than 3 states in a row; (2) x is initially negative for 0+ states, then non-negative forever.
▶ Show Solution
Solution (MOKA syntax)
1. G((x<0 & X x<0) ==> (X (X ! x<0))) 2. x<0 U (G ! (x<0))
Q4e — Past LTL (finite traces)
Using past LTL operators Y (yesterday) and H (historically), find the shortest finite trace satisfying: (1) a∧Yb, (2) Hb, (3) b∧H(b→Ya).
▶ Show Solution
Solution
a ∧ Yb → {b},{a} (length 2; yesterday of pos 1 is pos 0 = b) Hb → {b} (length 1; single state with b) b∧H(b→Ya) → {a,b},{b} ... (see lecture notes for full derivation)

CFL / RL Task 1 – Formal Languages (Coffee Machine) ⭐ ▾

Context: Four formalisms for a coffee machine — automaton A, regex E=(cd)*, LTL formula F=G(c→Fd), and grammar G: S→cd|cSd|SS. Questions compare languages, test ambiguity, regularity, and proof by induction.

⭐ Q1a
For each pair of models M1, M2 ∈ {A,E,F,G}, find a minimal-length word in L(M1)\L(M2), or show no such word exists.
▶ Show Solution
Solution (key differences)

L(A)=L(E): ε ∈ both; they accept exactly (cd)*. L(F): accepts "d" which A,E,G don't → F \ A = {d}. L(G): accepts ccdd (nested) which others don't → G \ A = {ccdd}. L(A) ⊆ L(F): A is a subset (ε is borderline — L(F) accepts ε by vacuous G). Full table has — where sets are equal.

⭐ Q1b — Grammar Ambiguity
Is grammar G: S→cd|cSd|SS ambiguous? If yes, fix it.
▶ Show Solution
Solution

Yes, ambiguous. String cdcdcd has two parse trees: either parse as (cd)(cd)(cd) using SS→S(SS)→..., or as S(cd) applied left-to-right. Fix by enforcing left-associativity:

S0 → S0 S1 | S1 S1 → cd | c S0 d
⭐ Q1c
Which models guarantee |w|_d ≥ |w|_c for all words w in their language?
▶ Show Solution
Solution

A: Yes — every accepting path repeats the loop reading d then c, so equal counts. E: Yes — L(A)=L(E). F: No — "ccd" satisfies G(c→Fd) vacuously but has more c's. G: Yes — by structural induction on derivations.

⭐ Q1d — Induction Proof
Prove the answer for grammar G from Q1c formally by structural induction.
▶ Show Solution
Solution Sketch

Base case: S → cd. |w|_d = 1 = |w|_c. ✓

Inductive case 1: S → cSd. By IH, for w derived from S: |w|_d ≥ |w|_c. Then |cwd|_d = |w|_d + 1 ≥ |w|_c + 1 = |cwd|_c. ✓

Inductive case 2: S → SS. By IH both halves satisfy property. Sum of two non-negatives is non-negative. ✓

Q1e — Regularity
Which of the four language models are regular?
▶ Show Solution
Solution

A: Regular (language of an ε-NFA). E: Regular (regular expression). F: Regular (any LTL formula over finite traces is expressible as a DFA). G: Non-regular — isomorphic to balanced-parentheses language via d↔(, c↔). Pump the word (cd)^n.

SEC Task 2 – Language-Based Security (Constant Time) ⭐ ▾

Context: Program P comparing two pairs of secrets (a1,a2) vs (b1,b2), outputting diff=0 or 1. Questions cover information flow analysis, non-interference verification, execution-time leaks, and constant-time reprogramming.

⭐ Q2a — Info Flow
Using information flow analysis, draw arrows between security levels {Alice, Bob, Dave, Trent} wherever a flow exists.
▶ Show Solution
Solution

Alice → Dave, Bob → Dave (both secrets flow into diff). No flow to Trent. No cross-flow Alice→Bob or Bob→Alice.

⭐ Q2b — Non-Interference
Provide a Floyd-Hoare triple formalising "Alice cannot learn anything about Bob". Does it verify?
▶ Show Solution
Solution
{a1_1=a1_2 & a2_1=a2_2 & diff_1=diff_2} [program copy 1]; [program copy 2] {a1_1=a1_2 & a2_1=a2_2}

This verifies in Chip. Alice fixes her variables and varies Bob's — but diff observed by Alice is the same regardless of Bob's secrets.

⭐ Q2c — Timing Side Channel
After instrumenting P with an execution-time counter "time", can Trent learn Alice/Bob secrets from "time"?
▶ Show Solution
Solution

Yes — timing leaks. Two runs: (a1=0,a2=0,b1=0,b2=0) ends with time=3; (a1=0,a2=1,b1=0,b2=0) ends with time=4. Different timing reveals whether a2=b2. Trent can distinguish.

⭐ Q2d — Constant-Time Algorithm
Design a constant-time program C (no boolean operators) that computes diff correctly and has the same execution time for all inputs. Verify with a Hoare triple.
▶ Show Solution
Solution (key idea)

Use arithmetic to avoid branching on secrets. Both branches of each if must have the same number of operations. Verify postcondition {time = 10}:

time := 0; diff := 1; time := time+1; if a1 != b1 -> time:=time+1; diff:=0; time:=time+1 [] a1 = b1 -> time:=time+1; diff:=1; time:=time+1 fi; if a2 != b2 -> time:=time+1; diff:=diff*0; time:=time+2 [] a2 = b2 -> time:=time+1; diff:=diff*1; time:=time+2 fi; diff:=diff-1; time:=time+2; diff:=-1*diff; time:=time+2 -- postcondition: {time = 10}
FM Task 3 – GCL Semantics Extension (split) ▾

Context: GCL extended with x,y := split(a1,a2) which assigns min to x and max to y. Questions formalise its semantics, characterise it with postconditions, and simulate it with standard GCL.

⭐ Q3b — Formal Semantics
Give the formal denotational definition of S⟦x,y := split(a1,a2)⟧(σ).
▶ Show Solution
Solution
S⟦x,y := split(a1,a2)⟧(σ) = if zi = A⟦ai⟧σ ∈ Z (i=1,2): σ[x ↦ min(z1,z2)][y ↦ max(z1,z2)] else: undefined
⭐ Q3d — Simulate with Standard GCL + Chip
Simulate x,y := split(2*x+1, x+y) using standard GCL. Provide a fully annotated command that verifies in Chip.
▶ Show Solution
Solution
{ x=N & y=M } u := 2*x+1; v := x+y; if u < v -> x := u; y := v [] !(u x := v; y := u fi { (u x=u & y=v) & (!(u x=v & y=u) }

Terminates (loop-free). Never gets stuck — guard and its negation always cover all memories.

SA / MC Task 4 – Sign Analysis & Program Properties ⭐ ▾

Context: A GCL program with two loops computing s = Σ2i and t = Σ4j³. Questions test Hoare triples, program graphs, sign analysis, full annotation, non-termination, and decidability of termination analysis.

⭐ Q4c — Sign Analysis Limits
For each of 5 statements about program values, answer: True, False, or Unknown from sign analysis alone.
▶ Show Solution
Solution
(i) t>=0 after init → True (sign analysis confirms 0 or +) (ii) s>0 after loop 1 → Unknown (0 is possible per analysis) (iii)s>=0 after loop 2 → Unknown (− possible per analysis) (iv) r>0 if N>0 → Unknown (abstract exec found, may not be real) (v) no exec if N≤0 → Unknown (abstract memory may not be realizable)
⭐ Q4e — Non-Termination
Show that C does not always terminate by providing a concrete non-terminating execution sequence.
▶ Show Solution
Solution

The second loop guard is m != N | n != N. If m≠N and n≠N initially, the loop body only modifies s (not m,n,N), so the guard stays true forever → infinite loop. Alternatively, use MOKA to model check ¬AF(terminated).

Q4f — Undecidability
Show (without Rice's theorem) there is no algorithm computing A(q) = {σ | ∃ execution ⟨q;σ⟩⇒*⟨q◀;σ'⟩} for every node q.
▶ Show Solution
Solution (Halting Reduction)

If such an algorithm existed, we could solve the Halting Problem: given program C and memory σ, run the analysis, then check σ ∈ A(q▷). This is true iff C terminates on σ. Since Halting is undecidable, no such algorithm exists.

FM 15% Exercise 1 – Semantics ⭐ ▾

Three program graphs PG1–PG3 given. Questions: does a GCL program generate each? Is each deterministic? How many complete execution sequences? Prove/disprove: deterministic+evolving ⟹ at least one complete execution. Extend GCL with Java-style for-loops.

⭐ Q1a — GCL from PG
For each PG, either provide GCL that generates it or argue why none exists.
▶ Show Solution
Solution

PG1: No — edge from final node back to initial is not allowed by GCL's edges function.

PG2: Yes:

y := 23; do x>0 -> x:=x+1 [] x<0 -> x:=x-1 [] x>0 -> y:=17; x:=x-1 od; if y=17 -> z:=13 [] x!=0 -> z:=7 fi

PG3: No — edge into final node is a Boolean guard alone, not an action.

⭐ Q1d — Deterministic + Evolving ⟹ Complete Execution?
Prove or disprove: every deterministic and evolving PG has at least one complete execution sequence.
▶ Show Solution
Solution

False. Counterexample: do true -> skip od. This PG is deterministic and evolving (skip is always enabled), but it loops forever and has no complete execution sequence.

⭐ Q1e — For-Loop Semantics
Define edges(q▷⇝q◀)⟦for(i:=a1; b; i:=a2){C}⟧ formally.
▶ Show Solution
Solution
edges(q▷⇝q◀)⟦for(i:=a1; b; i:=a2){C}⟧ = edges(q▷⇝q)⟦i:=a1⟧ ∪ {(q, b, q')} ∪ {(q, ¬b, q◀)} ∪ edges(q'⇝q'')⟦C⟧ ∪ edges(q''⇝q)⟦i:=a2⟧

Equivalent to: i:=a1; do b -> C; i:=a2 od

CFL 20% Exercise 2 – Formal Languages (API protocol) ⭐ ▾

Formalising correct API usage (open/write/read/close) with DFA, regex, grammar, and PDA. Language intersection, closure properties, and pumping lemma arguments.

⭐ Key Pattern
Recognise the pattern: given multiple equivalent formalisms (DFA, regex, grammar), compare their languages and determine regularity/non-regularity.
▶ Show Solution
Strategy

1. Compute a few sample words for each formalism. 2. Check containment by finding a witness word in L1\L2. 3. For regularity: DFA/NFA/regex → regular. Grammar with self-embedding / counting → use pumping lemma to show non-regular. 4. PDA accepts exactly context-free languages.

VER 20% Exercise 3 – Program Verification ⭐ ▾

Two fully annotated GCL programs to verify: one computing n³, one involving arrays. Questions require constructing loop invariants and reasoning about a new proof rule for array assignment.

⭐ Q3b — Annotated n³ Proof
Prove that the given GCL program computes n³ by constructing a fully annotated Floyd-Hoare triple.
▶ Show Solution
Strategy

Key pattern for proving power/polynomial loops: the invariant must track the current partial sum as a polynomial in the loop counter i, together with i's range. Work backwards from the postcondition to find the invariant, then verify each proof obligation (init, body, exit) manually or with Chip.

Ex 4–8 Exercises 4–8: Analysis · Security · CFL · Model Checking · Induction ▾
⭐ Ex 4 — Sign Analysis (10%)
Statements about program graph properties and sign analysis. Apply detection-of-signs analysis; identify True / False / Unknown answers.
▶ Show Solution
Key Point

Sign analysis is a sound over-approximation — if the analysis says "only +" it is truly always positive. If it says "0 is possible" the result is Unknown (may not be realisable). Never False directly from analysis.

⭐ Ex 5 — Language-Based Security (10%)
New security analysis; questions about non-interference, program security, and correctness/soundness of analyses.
▶ Show Solution
Key Point

Non-interference: a program is non-interfering if public outputs are independent of secret inputs. Information flow analysis is a sound over-approximation; it may flag flows that don't actually carry information. Distinguishing "flow exists" from "information is leaked" is critical.

⭐ Ex 6 — Context-Free Languages (15%)
CFG design, derivation trees, parse trees, ambiguity identification, and PDA construction.
▶ Show Solution
Key Point

Grammar ambiguity checklist: (1) does any nonterminal have two productions that can generate the same string? (2) Draw two different parse trees for the same string. Fix with stratification (precedence levels) or left/right associativity rules.

Ex 7 — Model Checking (5%)
LTL properties on a transition system. Determine which formulas hold.
▶ Show Solution
Key Point

Evaluate LTL on all paths from the initial state. G = all states on all paths; F = some state on some path; X = next step; U = until. Check each path in the system exhaustively for small systems.

Ex 8 — Induction Proofs (5%)
Structural induction proofs over programs or grammars.
▶ Show Solution
Key Point

Template: (1) identify the structure (program/grammar rule). (2) Base cases = atomic rules. (3) Inductive cases = composite rules — apply IH to sub-components. (4) Conclude for the whole.
